00 Executive summary

TaskAlert is a genuinely well-structured feature-sliced app for its size — repository/controller separation, a real dependency injection container, and a competent token-refresh interceptor are not things I see in every early-stage Flutter codebase. The gap between "this is a solid weekend-project skeleton" and "this can carry paying customers' data" is one authentication design flaw, an unset production build pipeline, and a handful of files that have grown past the point any one engineer can safely change them.

Two findings are release-blocking on their own: the OTP verification flow is architected so the API can (and during testing, does) hand the one-time code straight back to the client, which the client then displays — this defeats the purpose of two-factor sign-in. And logout only clears local storage; the refresh token is never revoked server-side, so a stolen device or a leaked token stays valid indefinitely after the user "logs out." Neither requires a rewrite to fix — see the checklist — but both must close before this reaches a production audience of paying customers.

Beyond those two, the codebase's biggest structural risk is size concentration: CreateRepetitiveScreen.dart is 5,649 lines, four other screens exceed 1,700 lines, and none of them have a single test covering them. That combination — huge stateful widgets, zero test coverage, no crash reporting — means regressions will be caught by users, not CI.

01 Project architecture

The lib/core/features/<feature>/{controllers,data/{models,repositories}} slice is applied consistently across nine features (auth, departments, employees, jobRoles, location, notifications, organization, taskInstance, tasks), each wired through GetIt in injection_container.dart. Repository interfaces are separated from their *Impl classes, which is exactly the seam you want for swapping the HTTP client or writing repository-level tests later. This part of the codebase would pass review at most shops.

The screens layer doesn't follow the same discipline. lib/screens/ holds 30 files with no further grouping, several of which are both "screen" and "everything that screen needs" — dialogs, form validators, dropdown builders, and API-calling logic all live in the same file as the widget tree.

Duplication

CreateOneTimeScreen.dart and CreateRepetitiveScreen.dart share the vast majority of their form-field, dropdown, and validation code — the recurring-schedule fields are the real delta. The dialog-opening helpers (openLocationFormDialog, openDepartmentFormDialog) are copy-pasted as free functions and re-imported into five different screens rather than extracted once into lib/components/. AuthRepositoryImpl repeats the same ~15-line "persist user session to secure storage" block three times (sign-up OTP, sign-in OTP, get-profile) with minor key differences — a single _persistSession(UserModel) helper would remove ~40 lines and the risk of the three copies drifting.

Dead code

lib/extras/MoreScreen.dart and lib/extras/NotificationScreen.dart are 16-line stub leftovers, superseded by the real implementations in lib/screens/ — they aren't imported anywhere and should be deleted. Commented-out print(...) lines are scattered through CreateRepetitiveScreen.dart, login_controller.dart, task_controller.dart, and MyTaskScreen.dart — harmless but noise that should be deleted, not commented.

Recommended structure change

Split each of the five largest screens into: a thin screen widget (scaffold + tabs/routing only), a widgets/ subfolder for the section widgets currently inlined as build methods, and move every dialog builder into lib/components/ as a proper reusable widget rather than a free function taking a BuildContext.

02 Security audit

OTP is returned by the API and rendered in the sign-in UICritical
lib/core/features/auth/controllers/login_controller.dart:55-58, 130-135
Problem
After requesting a sign-in OTP, the controller reads apiResponse.data['otp'] unconditionally and, if present, appends it to the success message shown to the user: "Your OTP is: $_otp". This is explicitly commented as a dev/test convenience ("Dev/test backends echo the generated OTP back in the response so the UI can auto-fill it"), but there is no build-flavor or environment gate around it in the client — it will render the code for any backend response that includes that field.
Impact
Two-factor/OTP verification exists specifically so that possession of a phone is required to sign in. If this ships as-is, and the field is ever present in a production response (misconfiguration, staging pointed at prod, a debug flag left on server-side), anyone who can trigger a sign-in request for a victim's phone number sees their OTP on-screen with no SMS access needed — a full authentication bypass.
Fix
Delete this behavior from the client entirely rather than gating it. OTP delivery is the backend's job (SMS/email); the client should never read or display an OTP value from a JSON body. If a dev convenience is truly needed, gate it behind kDebugMode && !kReleaseMode in the client and remove the field from the API response outside non-prod environments.
// login_controller.dart — remove this branch outright
final result = await _authRepository.signIn(phoneNumber: phoneNumber);
if (result is Success) {
  _currentPhoneNumber = phoneNumber;
  _successMessage = 'OTP sent successfully to $phoneNumber';   // no OTP value, ever
  notifyListeners();
  return true;
}
Logout never revokes the session server-sideCritical
lib/core/features/auth/data/repositories/auth_repository_impl.dart:628-631
Problem
logout() is await _secureStorage.deleteAll(); — nothing is sent to the backend. The refresh token (and the access token, until it naturally expires) remains valid on the server.
Impact
A device that's lost, stolen, or had its storage extracted before "logout" was pressed keeps a live, usable refresh token forever — logout provides zero security benefit against token theft, only a local UX reset. For a SaaS product handling employee/organization data, this is the difference between "we can revoke a compromised session" and "we can't."
Fix
Call a backend revoke/logout endpoint with the refresh token before clearing local storage, and clear local storage even if that call fails.
@override
Future<void> logout() async {
  final refreshToken = await _secureStorage.read(key: 'refresh_token');
  try {
    if (refreshToken != null && refreshToken.isNotEmpty) {
      await _httpService.post('/auth/logout', body: {'refreshToken': refreshToken});
    }
  } catch (_) {
    // best-effort — still wipe local state below
  } finally {
    await _secureStorage.deleteAll();
  }
}
No certificate pinning; API base URL hardcoded with no environment separationHigh
lib/core/network/dio_http_service.dart:13
Problem
baseUrl: 'https://task-alert-backend.onrender.com/api/v1' is a literal in the service constructor — there is no dev/staging/prod flavor mechanism (no --dart-define, no flavor-specific entry points), and Dio uses the platform trust store with no pinning.
Impact
Every build — including anything a developer sideloads — talks to the single production API, so there's no safe way to test against staging without editing source. Without pinning, a device with a malicious root CA installed (common on compromised or MDM'd corporate devices) can MITM every request, including the auth token exchange.
Fix
Move the base URL to String.fromEnvironment('API_BASE_URL', defaultValue: ...) fed by --dart-define-from-file per flavor, and add certificate pinning via Dio's HttpClientAdapter (e.g. dio_certificate_pinning or a custom SecurityContext) for the production flavor at minimum.
Release build ships without code obfuscation or shrinking configuredHigh
android/app/build.gradle.kts:54-65
Problem
The release build type only sets a signing config — there's no isMinifyEnabled/isShrinkResources, and the Flutter build isn't invoked with --obfuscate --split-debug-info anywhere in the repo.
Impact
Shipped APK/AAB is trivially decompilable — class names, the API base URL, storage key names, and business logic are all readable with a few minutes of jadx. Combined with no cert pinning, this materially lowers the effort needed to reverse-engineer and attack the API.
Fix
Enable R8 in the release build type and build with obfuscation: 
// android/app/build.gradle.kts
buildTypes {
    release {
        isMinifyEnabled = true
        isShrinkResources = true
        proguardFiles(getDefaultProguardFile("proguard-android-optimize.txt"), "proguard-rules.pro")
        signingConfig = if (keystorePropertiesFile.exists()) signingConfigs.getByName("release")
                        else signingConfigs.getByName("debug")
    }
}

// build command
flutter build appbundle --obfuscate --split-debug-info=build/symbols
A transient refresh failure wipes the entire local session, not just an expired oneMedium
lib/core/network/auth_interceptor.dart:53-61
Problem
_doRefresh() catches every error (network timeout, DNS failure, 500 from the backend) and returns null the same way it would for a genuinely expired/invalid refresh token. The caller then runs _secureStorage.deleteAll() unconditionally when newToken == null.
Impact
A user on a flaky connection whose access token happens to expire mid-request gets fully logged out and loses all cached profile/org state, indistinguishable from an actually-revoked session. At scale this shows up as "random logouts" support tickets.
Fix
Only wipe storage when the refresh call itself returns a definitive 401/403 from the auth endpoint; on network/5xx errors during refresh, surface a retryable error and leave stored tokens intact.
User session persisted as ~20 discrete secure-storage keys instead of one signed session objectMedium
lib/core/features/auth/data/repositories/auth_repository_impl.dart
Problem
Each of sign-up, sign-in, and get-profile writes 12-16 separate secureStorage.write calls (user id, email, phone, DOB, role, org id/name, avatar URLs, task permission…). flutter_secure_storage encrypts each value at rest, so this isn't a raw-plaintext exposure, but it means the three call sites can silently drift (get-profile already omits keys that sign-in writes, like user_gender/user_active_organization), and a partial write (app killed mid-sequence) leaves an inconsistent session with no way to detect it.
Impact
Stale/mismatched local user state (e.g. showing an old organization name after switching orgs) and a larger attack surface of individually-readable keys for anything with debugger/backup access to the keystore-backed store.
Fix
Serialize the full UserModel/ProfileModel as JSON under a single user_session key, write it atomically in one _persistSession() helper called from all three sites, and read it back through one deserializer.

03 Performance audit

ListView.separated(shrinkWrap: true) used for every list instead of lazy ListView.builderHigh
EmployeesScreen.dart (×4), LocationListScreen.dart, DepartmentListScreen.dart, OrganizationListScreen.dart
Problem
shrinkWrap: true forces Flutter to lay out and measure every item up front instead of building lazily as items scroll into view — it's the one flag that defeats the entire point of using a ListView over a Column. It's used consistently across the list-heavy management screens (employees, locations, departments, organizations), typically because the list is nested inside another scrollable.
Impact
Fine at 20-30 rows in a demo org. An organization with hundreds of employees — the exact customer this app is selling to — will jank on first paint of that screen and hold every row's RenderObject in memory at once.
Fix
Restructure so the list owns the scroll axis directly (CustomScrollView + SliverList if it must coexist with a header/filter bar) rather than nesting a shrink-wrapped list inside a scrollable parent.
God-widgets make targeted rebuilds impossibleMedium
Problem
With CreateRepetitiveScreen.dart at 5,649 lines as effectively one build method plus helpers, any setState call — e.g. toggling one checkbox — re-runs the build for the entire form, dropdowns, date pickers and all.
Impact
Frame-time cost scales with the whole screen's widget count on every keystroke/toggle, not with what actually changed — the classic cause of typing jank in large forms.
Fix
Extract independent sections (schedule picker, proof-type selector, reporting-list picker) into their own StatefulWidgets with their own local state, so a change in one doesn't rebuild the others.

Const usage is present but inconsistent — spot checks show HomeScreen.dart and CreateRepetitiveScreen.dart already use const in hundreds of places, so this is a matter of running flutter analyze with prefer_const_constructors enabled and fixing stragglers rather than a systemic gap.

Images go through cached_network_image, which is the right call for avatar/proof-photo thumbnails and already handles disk caching — no action needed there beyond confirming a memCacheWidth/cacheWidth is set for large proof photos so full-resolution images aren't decoded just to render as thumbnails.

Isolates are not used anywhere, which is fine today — nothing in the sampled code (JSON decoding of list responses, image handling) is heavy enough to justify it yet. Revisit if task-instance list responses grow past a few hundred KB.

04 Flutter best practices

Disposal hygiene, where checked, is correct: SignInScreen and OtpVerificationScreen both pair addListener/removeListener on their ChangeNotifier controllers and dispose their TextEditingControllers and FocusNodes correctly. 29 dispose() overrides exist across 24 files against 20 files that hold a disposable controller — worth a full pass specifically on the five largest screens, since a missed removeListener on a GetIt-singleton controller (which outlives the screen) is a real, easy-to-introduce leak in files that size.

di.init() is not awaited, and runs before WidgetsFlutterBinding.ensureInitialized()Medium
lib/main.dart:13-17
Problem
void main() { di.init(); WidgetsFlutterBinding.ensureInitialized(); runApp(...); } — the binding is initialized after DI setup runs, and the async init() call isn't awaited at all.
Impact
Harmless today only because init() happens to contain no await internally, so it completes synchronously. That's an accident of the current implementation, not a guarantee — the moment anyone adds an awaited call inside init() (a remote-config fetch, a plugin that needs the binding), this silently breaks in a way that's hard to reproduce.
Fix
Future<void> main() async {
  WidgetsFlutterBinding.ensureInitialized();
  await di.init();
  runApp(const MyApp());
}

Error handling is centralized well through NetworkException.fromDioError, which maps Dio's exception types to a small, user-safe NetworkErrorType enum and prefers the backend's own message when present — this is a genuinely good pattern and every repository method routes through it consistently.

Null safety is enabled project-wide (SDK ^3.8.0) with no // ignore: escape hatches spotted in the sampled files.

05 State management

There's no Provider/Riverpod/Bloc/GetX dependency in pubspec.yaml — state management is hand-rolled: every feature controller extends ChangeNotifier, is registered in GetIt (singletons for auth-adjacent controllers, factories for the CRUD-feature controllers), and screens call sl<XController>() then manually addListener/setState. It works, and it's honest about what it is — but it puts the correctness burden (remembering to remove the listener, not over-notifying) entirely on each screen author rather than a framework enforcing it.

The DI split between registerLazySingleton (repositories, auth-related controllers) and registerFactory (CRUD feature controllers) is a reasonable lifecycle choice — factories mean each screen gets a fresh controller instance rather than sharing mutated list/filter state with a previous screen instance.

Recommendation: not urgent enough to block release, but worth budgeting for once the team grows past ~2-3 engineers: standardizing on Riverpod (keeps the existing repository/controller shapes, just replaces GetIt+ChangeNotifier's manual wiring with compile-time-safe providers and automatic disposal) removes an entire class of "forgot to remove a listener" bugs that will otherwise scale with headcount.

07 Database review

Not applicable in the traditional sense — there's no Firestore, SQLite, Hive, or Isar in this codebase. All feature data is fetched fresh from the REST API on each screen visit; shared_preferences and flutter_secure_storage hold only session/profile state, not domain data. This is a legitimate architectural choice for an internal-tools-style app and removes an entire category of local-database-security concerns (no security rules, no local injection surface to audit) — but it also means there's currently no offline capability and no local caching layer, which will matter for the SaaS-scale conversation in §14.

08 Authentication

The flow is phone-number-first and OTP-verified for sign-in (passwordless), while sign-up collects a password in addition to OTP verification, and there's a separate updatePassword endpoint — meaning password and OTP are both live credential mechanisms depending on the flow, which is a slightly unusual hybrid worth confirming is intentional product design rather than incremental drift.

  • Session persistence / auto-login: main.dart checks for a stored auth_token at startup and routes straight to HomeScreen if present — correct pattern, though it trusts token presence, not validity; an expired-but-still-stored token will bounce through one failed request and the refresh flow before the user notices, rather than being caught at startup.
  • Refresh tokens: handled server round-trip via /auth/access-token, stored in flutter_secure_storage — correct storage choice. See §2 for the revocation-on-logout gap.
  • Biometric auth: not implemented — no local_auth dependency. Reasonable to defer, but worth offering as an option before enterprise customers ask for it, since it's the cheapest lever to reduce OTP/SMS friction on repeat sign-ins.
  • OTP verification: see the critical finding in §2 — this is the item that actually blocks release, not the hybrid model above.

09 UI / UX review

This review was performed by reading source, not by running the app in a device/simulator — I can speak to patterns visible in code (loading/error state wiring, responsive units, theming) but cannot confirm actual rendered layout, overflow, or keyboard behavior on-device. Treat this section as a code-level signal, not a substitute for manual QA on real screen sizes before release.

  • flutter_screenutil is used for responsive sizing app-wide with a fixed designSize, and main.dart explicitly locks textScaler to 1.0 — this overrides the user's OS-level font-size accessibility setting for the entire app, which is a real accessibility regression for low-vision users and should be reconsidered (clamp to a sane range like TextScaler.linear(scale.clamp(0.9, 1.3)) rather than pinning to exactly 1.0).
  • Controllers expose isLoading/errorMessage/successMessage getters consistently, which is the right shape for screens to drive loading spinners and error banners — whether every screen actually surfaces all three states (versus silently swallowing an error) wasn't fully verifiable without running the UI.
  • No dark mode: ThemeData(textTheme: GoogleFonts.interTextTheme()) is the only theme configuration — colors are hardcoded per-widget (e.g. Color(0xFF0A0258) constants repeated across MoreScreen/ProfileSetting) rather than pulled from a ColorScheme, so there's no dark-mode support and no single place to re-theme the app.

12 Code quality

  • File naming: screens use PascalCase filenames (MyTaskDetails.dart, HomeScreen.dart) against the Dart/Flutter convention of snake_case file names (which the core/features/ tree correctly follows) — an easy, mechanical, high-value cleanup since flutter_lints' file_names rule would catch this if enabled.
  • Typo: employee_respository_impl.dart (repository misspelled) — harmless but worth fixing while nothing external depends on the filename.
  • Dead code: lib/extras/ stub files (§1), commented-out debug prints throughout, and a fully commented-out duplicate getProfile() override left at the bottom of auth_repository_impl.dart (lines 494-517).
  • Long methods / large widgets: covered in §1 and §3 — this is the dominant code-quality issue in the repo by volume.
  • Comments: where present, comments in the network/auth layer are unusually good — they explain why (e.g. the separate refresh-Dio instance to avoid recursive interceptor loops), not just what. That standard isn't consistently carried into the screens layer.

13 Testing

Zero real test coverageHigh
Problem
test/widget_test.dart is the only test file in the repository, and it's the default Flutter counter-app template test — it does not exercise this app at all.
Impact
Every one of the god-widget screens in §1, the token-refresh logic in §2/§6, and every repository's JSON parsing is entirely unverified by automation. Regressions are caught by users in production, not CI.
Recommended minimum before scaling the team
  • Unit tests for every *RepositoryImpl (mock HttpService, assert on success/failure mapping) and every controller's state transitions — these are the highest-value tests per hour spent given the repository/controller split already in place.
  • Unit test specifically for AuthInterceptor's refresh-coalescing logic — it's the most intricate piece of logic in the app and the easiest to silently break.
  • Widget tests for the sign-in/OTP/sign-up flow given its security sensitivity.
  • Integration test for the create-task flows (one-time and repetitive) given their size and complexity.

14 SaaS readiness

The data model already has real multi-tenancy shape — organization vs. activeOrganization fields on the user, an organization-switch dialog, department/location/job-role scoping — this is not a bolt-on, it looks designed in from early on, which is the hard part to retrofit. What's missing is the operational layer around it:

  • No monitoring/observability: no crash reporting (§10) and no API-level tracing means the team is blind to production incidents until a customer reports them — a hard blocker for supporting "thousands of concurrent users" with any confidence.
  • List rendering won't hold at enterprise scale: the shrinkWrap pattern (§3) is specifically the kind of thing that works in every demo and QA org (small headcount) and fails at the first genuinely large enterprise customer's employee list.
  • Licensing risk under a paid-SaaS model: Syncfusion's date picker (§11) needs a revenue/headcount check now, before it's a surprise during a compliance or procurement review from an enterprise customer.
  • Session revocation (§2) matters more, not less, at SaaS scale — an admin needs to be able to kick a departed employee's device off the account, and today nothing server-side is even called on logout to build that on top of.

None of this requires an architecture rewrite — the repository/controller/DI foundation is the right shape to build on. It requires closing the auth gaps, adding observability, and doing the list-virtualization fix before selling into an organization large enough to feel it.
